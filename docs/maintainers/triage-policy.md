# Triage policy

Use `needs-triage` for unreviewed work. Confirm reproducibility, platform, affected contract, safety implications, and maintenance scope before accepting an issue.

Priority guidance:

- `priority/critical`: security issue, data corruption, unsafe claim, or release-blocking regression.
- `priority/high`: broken core scenario, cross-platform failure, or contributor-blocking defect.
- `priority/normal`: accepted feature, documentation gap, or bounded technical debt.
- `priority/low`: optional polish or speculative extension.

Close unsupported requests politely with the architectural or maintenance reason. Convert significant accepted changes into design proposals and ADRs.
