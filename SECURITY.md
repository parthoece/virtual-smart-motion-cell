# Security policy

## Supported versions

| Version | Supported |
|---|---:|
| Latest minor release | Yes |
| `main` | Best effort |
| Older releases | No |

## Reporting

Do not open a public issue for a suspected vulnerability. Use GitHub private vulnerability reporting after the repository is published, or contact `SECURITY_CONTACT@example.com` until that feature is enabled.

Include affected versions, impact, reproduction, and suggested mitigation when possible. Maintainers aim to acknowledge complete reports within five business days.

## Scope

Security reports may cover command authorization, path traversal, unsafe configuration, secret exposure, dependency compromise, WebSocket/API issues, release integrity, and adapter isolation. Functional-safety certification is outside project scope.
